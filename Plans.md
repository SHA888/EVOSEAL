## Phase 2: P1 — Safety Hardening & Integration

Scope: Establish threat model, decide on isolation strategy, build testable infrastructure,
and establish cost visibility. P0 is complete; all items below are prerequisites before P2
(architecture docs, dashboard) work begins.

**Execution strategy:** 2.1–2.2 first (establish ground truth for safety decisions).
Then 2.3–2.7 (safety tests + testability infrastructure) in parallel with 2.8–2.11
(cost tracking + doctor). All tasks unblock P2 work.

## Phase 3: P2 — Documentation, Dashboard, Test Coverage & Evolution Archive/Rollout

Scope: Close the P2 gaps identified against `TODO.md` P2 (lines 87-129): extend the existing
Phase 3 bidirectional-evolution doc rather than rewrite it, add cost/diff/checkpoint-loading
capabilities to the dashboard, fill real coverage gaps in `core/` tests (not padding already-tested
modules), and introduce a structured evolution-archive format with an MVP-scoped rollout-gating
mechanism (full candidate/beta/stable automation deferred — see 3.11/3.12 note). Spec skip reason:
no external API consumers of the archive/rollout formats yet; ADR-level design docs (3.10, 3.11)
are the DoD instead of a formal spec.md.

**Execution strategy:** 3.1-3.2 (doc gap-fill) can proceed immediately and independently.
3.3-3.5 (dashboard) depend on nothing new but 3.5 (checkpoint-only load) should follow 3.10
(archive format) if it wants to read the new structured units; otherwise it can load raw
checkpoints and be revisited later — sequenced as no hard dependency, but noted. 3.6-3.9
(test coverage) are independent of each other and of the dashboard work. 3.10 (archive format)
should land before 3.11 (rollout gating) since gating needs a place to record stage transitions.
